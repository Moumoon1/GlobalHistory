# Data Sources

The production database should be curated rather than copied wholesale from a single book or a massive knowledge graph.

## Recommended Source Layers

1. **Editorial scope**
   - Use broad global-history reading notes as the outline.
   - Rewrite every entry in original wording.
   - Store page or chapter references privately if you use a physical or legally owned ebook copy for study.

2. **Open reference checks**
   - Use open educational resources and encyclopedia-style references to verify dates, names, and regional framing.
   - Do not paste long passages into the app database.

3. **Primary source context**
   - Use primary-source collections for source links and context around specific periods.
   - These are useful for later detail pages, not for bulk event generation.

4. **Map data**
   - Keep modern-country boundaries from Natural Earth or another open map provider.
   - Historical borders should be optional overlays later, not a blocker for the current product.

5. **Images**
   - Prefer Wikimedia Commons or museum/open-access collections.
   - Store image URL, caption, creator if known, license, and source page URL.

## Curation Rule

Every event should have:

- one concise original description,
- at least one verification source before public launch,
- an internal importance score,
- a mapped region or macro-region,
- a time range that can be queried by the app.

The seed table is allowed to start with general knowledge and editorial judgment. Public launch data should be checked source by source.
